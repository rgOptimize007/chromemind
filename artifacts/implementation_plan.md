### 2. Git Automation Agent

The `GitAgent` will be the strict, sole point of interaction with Git and GitHub. The `OrchestratorAgent` will rely on it to perform codebase maintenance.

#### [NEW] `agents/git_agent.py`
The `GitAgent` class will abstract all Git behaviors:
- **Branch Management:** Executes local Git CLI commands (`git checkout -b feature/...`) branching off from `main`.
- **Commit Management:** Formulates structured, contextually rich commit messages describing changes.
- **Push & Squash Logic (via GitHub MCP):** If there are more than 3 commits pending a push, it mechanically squashes them (`git reset --soft HEAD~N` / `git commit -m ...`) before invoking the push.
- **PR Conflict Handling (via GitHub MCP):** Periodically polls or evaluates PR updates over MCP. If conflicts are reported, `GitAgent` interrupts the orchestrator sequence, emits an actionable log alert, and pauses for your supervision.

#### [MODIFY] `agents/orchestrator.py`
- Import and utilize `GitAgent` directly. The orchestrator handles the high-level intent (e.g., "save changes"), but delegates all actual checkout, commit, and sync commands exclusively to `GitAgent`.

---

### 3. Fetching Data for Tab Groups (Chrome Extension)

While the ChromeDevTool MCP or standard CDP endpoints provide raw access to tabs, they do not inherently expose `tabGroups`. An extension is the most reliable bridge.

#### [NEW] `extension/manifest.json`
- Define a Manifest V3 extension with `tabs` and `tabGroups` permissions.

#### [NEW] `extension/background.js`
- Set up an endpoint or polling mechanism (like a local WebSocket or saving to a known local JSON file location) that reads `chrome.tabGroups.query()` and `chrome.tabs.query()`.

#### [MODIFY] `skills/scrape_tabs.py`
- Redirect the script to read the local state generated by the custom Chrome Extension, allowing full `tab_group_name` capture alongside tab URLs.
